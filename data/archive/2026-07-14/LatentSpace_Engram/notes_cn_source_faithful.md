## 一、研究导读：核心命题

- Engram 的赌注不是简单做更好的 RAG，而是：随着企业内部 agent 生成海量代码、文档、演示和操作轨迹，企业知识库可能在约 **18 个月**内扩张到万亿 token 级别；每次让前沿模型从头读取，即使上下文窗口足够长，也会遭遇成本、KV cache、context rot 和整体推理准确率问题。
- 公司提出 **cartridge（知识盒/知识胶囊）**：先让模型对某个语料库“学习、自己提问、做测验、解决问题”，再通过梯度下降把知识压缩到可装卸的参数/权重状态。Biderman 声称这种表示可比原始文本压缩约 **1000 倍**，并让模型用更少 token、更少 prefill、更低混乱度完成任务。
- 他不主张权重记忆替代文本。理想系统同时拥有：可审计的笔记/RAG、上下文压缩、路由，以及类似人类神经系统的参数化直觉。Chef 既需要菜谱和实验日志，也需要手感、味觉和不必每次重读菜谱的内化技能。
- 商业落点优先在企业：Harvey 这类法律工作流拥有高密度、难检索、需整体理解的知识。长期愿景则是每个人拥有自己控制的权重/适配器，模型随着使用持续变得“更适合这个人”，最终部分个性化推理可能在个人设备运行。
- 基础设施含义很重：若每个人、团队都有独立适配器，就需要管理数百万参数端点，在磁盘与 HBM 间高速装载、交换、更新；这不是单一算法产品，而是新的训练—存储—推理调度层。

## 二、按原始流程的深度纪要

### 1. 01:45–07:12｜从以色列特种作战、计算神经科学到创办 AI 研究公司

Biderman 回顾，其职业路径事后看连贯，实时经历却并非预先规划。他在特拉维夫长大，曾任海军特种作战军官。那段经历很像创业：寻找看似疯狂但可行的任务，研究可行性，向组织争取预算与人员，再通过高强度执行把设想变成现实。

退役后，他在以色列学习认知神经科学，对大脑、人类行为和统计产生兴趣；之后到纽约攻读计算神经科学博士。这个学术群体在神经网络尚未成为产业热点前几十年就一直研究相关问题，结合统计、物理、生物来理解神经网络、动物行为和大脑。

ChatGPT 出现后，他决定深入 LLM：先在 Mosaic 工作并参与 LoRA 相关研究，后来到 Stanford 与 Chris Ré、Scott Linderman 合作；现公司的其他联合创始人来自 Stanford、Cornell、Berkeley 等研究网络。

主持人问，以色列特种部队为何持续产生 Wiz 等创业者。Biderman 不把它神化为“军队训练创业”：关键是在 18 岁就承担真实责任、接触成年人、争取资源和预算，不再只是为考试做项目，因此会提前训练社会能力、成熟度和团队协作。他也强调这是 package deal，服役耗时且有代价，并不建议所有国家照搬。

他认为以色列社会给人多次“重新发牌”：高中不突出，可能在军队找到位置；军队一般，仍可在大学发现科学天赋；因此一次早期失败不会永久封闭路径。他自己 24 岁才进大学，进入美国博士项目时年龄更大，但此前经历也带来成熟度。

### 2. 07:12–10:28｜问题起点：效率、半监督学习与让模型“预先学习”语料库

他的博士研究聚焦当时不流行、但他认为会重新重要的**半监督学习**：没有万亿 token，而只有很少样本时，怎样高效外推。其论文常有同一种图：横轴是成本/资源，纵轴是准确率，寻找 Pareto frontier。这个训练塑造了他“用更少资源做更多事”的研究直觉。

在 Chris Ré 实验室，他研究 agent 和 Minions 类思路：让子 agent 经济地处理大型知识语料。随后团队继续追问，除 agent orchestration 外，能否利用训练本身，使模型不必每次重新读完大量 token。

多个联合创始人并行形成相似思路：

1. 给模型一个很大的知识库；
2. 不立即提问，而是让模型提前“学习”——自我提问、自我测验、尝试解决问题；
3. 再像预训练一样用梯度下降，把学习结果压缩为参数状态；
4. 形成可加载、卸载的 **cartridge**，代表某个语料库、任务或技能。

Biderman 把 cartridge 描述为模型的“脑状态”：相对原始上下文可压缩约 1000 倍；加载后模型不必重新消费全部 token，理论上更少困惑、更准确、更经济。

### 3. 10:29–14:15｜Chef 类比：RAG 是菜谱，参数化记忆是厨师的神经系统

Cartridge 可以是 corpus-specific，例如某企业全部内部文档；也可以是 task-specific，例如某种技能。

他用现场做菜解释文本表示的边界：拥有 Thomas Keller 等人的顶级菜谱，不等于真正拥有顶级厨师能力。当前 LLM 像每次第一次进厨房，边读教科书边机械测量、执行；它缺乏厨师抓盐、揉面、尝味以及根据经验创造新菜的直觉。

主持人问，这与 RAG 从菜谱检索最相关片段有何不同。Biderman 明确说他不押注“笔记不再需要”：顶级厨师也写菜谱、日志，记录实验成功和失败；文本表示可解释、可追踪，仍然重要。但厨师还有大脑、手、舌头和内化经验，可以不每次重读同一页，并能在原菜谱之外创新。

因此目标是二者结合：

- **外部化知识**：文档、wiki、日志、RAG，便于审计和精确回溯；
- **内部化知识**：数字/参数形式的关联、直觉和技能；
- 只让知识工作者写笔记而每天抹掉其大脑，会严重损害工作；只凭记忆而不写笔记也同样低效。

### 4. 14:16–18:00｜为什么企业数据会快到“互联网规模”，长窗口仍解决不了

Biderman 认为市场低估 agent 对企业知识空间的放大：agent 会持续生成代码、文档、演示、分析和操作记录。约 18 个月后，真正 AI-native 的公司可能拥有万亿 token 级别的内部专有数据。两三年前在 Mosaic，这已被称作互联网规模的预训练数据；未来单个企业可能达到类似数量级。

即使今天还能用文本 wiki 和索引管理，规模扩大后会出现：

- 如何持续把万亿 token 压成及时更新的知识库；
- 索引本身会有多大；
- 一个对公司一无所知的前沿模型，每次处理这些知识要消耗多少 token 和成本；
- 更关键的是，模型读得越多不必然懂得越多，context rot 会让准确率下降。

他预计这种问题即使上下文扩大到 **1000 万 token**仍存在。能把 token 塞进窗口，只代表接口没有报错，不代表模型能对全部信息进行整体、稳定、因果一致的推理。

### 5. 18:01–22:40｜长上下文、compaction 与 neural memory trace 各自解决什么

主持人提出反方：企业数据也许没大到必须写入权重；RAG、便宜开源模型、上下文压缩是否足够？Biderman 承认，研究界很难构造一个“只有 in-weight training 能做、in-context learning 必然失败”的纯理论例子，因为反方总可以假设下一代模型有 1000 万甚至无限窗口。

他把 continual learning 和 memory 看作“伪装的长上下文问题”：如果模型真的能看见全公司数据，限制主要有两类。

**第一，context rot。** 输入越长，模型越容易混淆，不能保证整体推理质量。

**第二，资源效率。** 即便理论上答得出来，每次 prefill 海量上下文仍浪费计算和显存。

Compaction 会让模型自行管理上下文、丢弃部分 token、保留部分 token。Biderman 认为它肯定是解决方案的一部分，而且每天进步；但它天然有损，且目前通常是较离散的“保留/丢弃”。会话足够长后，模型仍可能忘记早期关键事实或把压缩后的摘要当成完整事实。

Engram 设想另一部分是 **neural memory trace**：同样有损、同样选择性遗忘，但不是保存成文本摘要，而是保存到参数/权重表示。

公司目前先解决较窄问题：让模型对大语料用更少 token、更少资源、更少混乱地推理。更长期才是让模型在科学、工程、国防等长期任务中边做边通过梯度更新——可被称为 test-time compute、test-time training 或持续学习的不同变体。

### 6. 22:40–24:20｜系统瓶颈：一篇 Wikipedia 文章为什么能占约 80GB“脑状态”

Biderman 用 Llama 70B 做量级类比：

- 一篇 Wikipedia 文章可能只有几十 KB；
- 让模型把它放入上下文后，其 KV cache/运行时 brain state 可能占约 **80GB GPU HBM**；
- 模型全部参数以 FP16 表示约 **140GB**量级，却能以有损方式编码接近整个互联网的统计知识；
- 结果是一篇 Taylor Swift 文章的运行时状态，与“整个模型参数所代表的世界知识”处在相近显存数量级，极不高效。

他把这称为 KV cache monstrosity，芯片、kernel 和软件团队都在努力解决。Engram 的另一种系统解释是：**把反复 prefill 的读取工作提前转化为训练计算**。某个时间点先把语料压进 cartridge，推理时加载后直接 decode，或只需要很短 prefill。

这与数据中心把 prefill 和 decode 解耦、放到不同专用卡上的趋势相容：Engram 不是简单让每次 prefill 更快，而是试图减少“每次都要 prefill”的需求。

### 7. 24:20–27:02｜Harvey/法律案例：RAG 很难回答“哪些交易没有完成”

主持人要求具体企业案例。Biderman以法律/投行文件系统为例：大量客户事项涉及融资、并购、借贷和交易材料。Engram 与 Harvey 合作处理非常大的文件系统。

许多真正有价值的问题不是搜索一个存在的句子，而是寻找**不存在或未闭环的状态**。例如：“今年哪些 M&A 交易还没有完成？”

要回答它，agent 必须：

1. 逐个查看客户事项；
2. 阅读相关文件和进度；
3. 判断哪些交易已形成闭环；
4. 对那些没有任何文档明确写着“未完成”的事项做负向推断。

RAG 很难搜索“没有出现的完成事件”。前沿模型加 compaction 可以暴力完成，但一次看似普通的企业查询可能花费**数千美元**，而熟悉全局的员工直觉上可以回答。

这类“整体大于部分之和”的问题正是训练式知识表示的目标。预训练之所以在整个 Web 上训练，而不是仅给模型一个全网 RAG 目录，是因为参数学习能建立跨文档关联、概括、插值和外推。Engram 想把同样机制缩小到企业或个人知识域。

### 8. 27:02–29:17｜长期产品形态：每个人拥有自己的权重

主持人问，是否为每家公司做 LoRA 式参数高效微调。Biderman说短期确实会利用 LoRA、cartridge、memory layer 等参数高效方法，但长期愿景更个人化：

- 每个人拥有一个模型、模型的一部分或一组权重，代表个人知识和专业能力；
- 与模型相处越久、授权的数据越多，模型越懂这个人；
- 这组权重由个人控制，而非只给通用提供商贡献反馈；
- 用户像养 Tamagotchi 一样有动力“喂养”它，因为收益直接回到自己的模型；
- 极长期，个人设备可能运行这些个性化部分。

他称新一代个人电脑的硬件正接近运行“接近万亿参数模型推理”的能力（这里可能是稀疏/MoE 或特定量化口径，需等待官方技术文档，不宜按稠密模型显存直接理解）。

短期仍优先企业，因为企业拥有高密度专业知识，是人工作时间最多、AI 使用最真实的场所。Engram 计划把参数记忆与可审计的上下文管理结合，而不是让不可解释权重单独承担全部事实。

### 9. 30:02–34:20｜什么应写入权重、什么应保留为文本？遗忘本身是必要能力

Biderman承认这是公司和整个记忆科学的开放问题。人脑也必须区分内部化与外部化：记住见过的一切并不一定是优势，超强自传式记忆可能带来干扰甚至痛苦；适度遗忘是健康的。

直觉上的分配原则包括频率、显著性和可供性：今晚酒店房号不如家庭地址或伴侣电话值得长期内化。但手工写规则会变成 whack-a-mole：不同企业、员工、任务的高价值信息完全不同，无法靠静态启发式决定进出。

他的“holy grail”是让模型自己学习：

- 同时拥有 notebook/文本笔记和参数化 associative brain；
- 判断哪些信息应从脑中直接调用，哪些应查笔记；
- 用户可明确说“记住/不要记住”，系统应尊重；
- 但不能依赖用户逐步监督，否则产品不可用。

与通用聊天产品的 thumbs up/down 不同，Engram 希望用户知道反馈会直接触发针对自己的额外计算和练习，让模型对这个用户变好，而不是不确定地进入下一版全局模型训练。

同时，用户反馈并非 ground truth。用户会自信地说错，未来模型在某些领域比用户更强，因此系统还须判断哪些反馈值得学习、哪些应忽略。这涉及目标函数和自主学习机制，仍需研究突破。

### 10. 34:20–37:57｜衡量智能的新角度：不是堆更多 token，而是用更少能量完成更难任务

公司暂未公开全部结果，计划未来数周/月逐步披露。Biderman愿意公开的主线是 **token efficiency**：模型能否更快找到需要的信息、用更少能量解决更难问题。

主持人问到模型路由：便宜模型不一定真便宜——若小模型反复尝试耗掉 100 美元，而强模型一次解决，后者反而更经济。Biderman同意路由必不可少：许多任务不需要最强模型，但高难任务必须知道何时升级。

Engram 不认为一个个性化模型会取代全部模型、更不认为因此可以“关闭 Stargate”。系统会是多模型的：

- 个性化/企业记忆层像长期员工或最了解你的朋友，始终观察上下文；
- 它知道该去哪里、应聚焦什么；
- 遇到高难任务时，把更精确、更富上下文的问题路由给顶级模型，让顶级模型工作数小时或数天；
- 真正困难在于模型版本持续变化，如何按正确时间、成本、准确率选择正确模型仍未解决。

### 11. 38:07–43:00｜团队构成与研究公司的产品纪律

团队高度专业化于记忆、持续学习、把知识写入权重，因此从普通意义看并不“多元”——很多人都是研究者；但能力倾向互补：有人偏数学和统计，有人偏系统，有人研究人机闭环和 AI 领导行为。

组织方式是把有多年直觉的博士/资深研究者，与刚进入领域、思路新鲜且执行力强的年轻研究者配对。前者帮助避开反复验证过的兔子洞，后者避免团队被既有范式锁死。

公司在 2025 年秋冬前沿实验室抢人和 AGI 焦虑最强时成立。联合创始人从一开始就强调，这不能只是 research club：缺的不是更多论文，而是产品；产品必须被分发，只有卖出用户真正喜爱的东西，才赢得继续做研究的权利。

### 12. 43:01–46:55｜真正的基础设施工作量：数百万适配器如何在磁盘与 HBM 间流动

招聘方面，公司需要两大类人：

1. **LLM performance / research engineers**：解决持续学习时如何不破坏基础模型、如何控制成本、该学哪些数据，并把模型跑快；
2. **基础设施、API 和数据库工程师**：若每个人/团队都有参数高效适配器，就会产生数百万独立端点，存放在不同位置，需从磁盘高效加载到 HBM，在推理时快速切换、更新和持久化。

字幕提到团队中有来自 Databricks、参与 vLLM 的推理负责人，但姓名识别可能不准，应以官网为准。

Biderman强调，如果方案成立，计算足迹不会因为“效率”而变小到无关紧要，反而会因任务边界扩张而非常大。提高效率不是做低端廉价产品；**效率与智能不可分离**。当前 scaling 范式是 more with more，已经走得很远也会继续；下一范式会加入 more with less，因为只有单位任务更高效，系统才能承接更长 horizon、更高难度的科学和工程任务。

## 三、投资映射：Engram 可能处在哪一层

### 1. 产品层

不是普通向量数据库，也不只是企业搜索。其差异化主张是把企业/个人语料预先训练成可装卸权重，并与 RAG、compaction、路由组合。近期可卖点应是成本、延迟和长语料整体问题准确率；远期才是自主持续学习。

### 2. 数据与护城河

潜在数据资产来自：企业语料、用户反馈、任务完成轨迹，以及“什么值得内化、什么值得检索”的学习信号。真正护城河要看 cartridge 是否能跨模型迁移、是否在模型版本升级后保持价值，以及能否证明相对 RAG/长上下文的稳定收益。

### 3. 基础设施与供应链

如果每位用户拥有独立 adapter，需求会从“少量大模型常驻显存”转向“大量小参数状态频繁装载”。这可能拉动：

- 高带宽内存与更高效 KV/权重管理；
- prefill/decode 分离；
- adapter caching、NVMe—DRAM—HBM 分层；
- 多租户隔离、加密、权限、版本和删除权；
- 训练与推理混合调度。

### 4. 主要待验证风险

- **技术可证伪性**：Biderman承认很难证明某任务只有 in-weight learning 能做；更强上下文、compaction 和 RAG 可能持续缩小差距。
- **灾难性遗忘与错误固化**：如何持续更新而不破坏已有能力，如何避免把用户错误反馈写进权重，是核心未解问题。
- **审计与合规**：参数记忆比文本难解释、难逐条删除；企业是否接受权重中潜藏敏感事实，需要清晰数据治理。
- **更新频率与成本**：若企业知识快速变化，cartridge 多久重训一次、旧状态如何失效，将决定经济性。
- **模型依赖**：专用适配器能否跨基础模型版本复用，还是每次前沿模型升级都需重做。
- **数字需复核**：约 1000 倍压缩、80GB KV 状态、万亿 token 企业语料及个人设备万亿参数推理均为节目口述或字幕转录，应等待论文、benchmark、产品文档和官方融资材料验证。

## 四、一句话结论

Engram 的核心押注是：AI 时代真正稀缺的不是更长的“临时工作记忆”，而是能把企业和个人经验持续内化、按需装卸、又与可审计文本协同的“长期参数记忆”；若成立，它将同时是模型算法、企业数据产品和新型推理基础设施问题。
